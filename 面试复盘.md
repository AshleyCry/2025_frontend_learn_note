## 20250710 智算科技一面

### SSE（Server-Sent Events）服务端推送

SSE是一种允许服务器向客户端实时推送数据的Web技术，基于HTTP协议，提供了一种简单高效的服务器到客户端的单向通信机制

服务端推送的背景与需求主要基于以下几个诉求：

- 实时通知：在很多情况下，用户期望实时接收到应用的通知，如新消息提醒、商品活动提醒等
- 节省资源：如果没有服务端推送，客户端需要轮询的方式来获取新信息，会造成客户端、服务端的资源损耗。通过服务端推送，客户端只需要在收到通知时做出响应，大大减少了资源的消耗
- 增强用户体验：通过服务端推送，应用可以针对特定用户或用户群发出有针对性的内容，如优惠活动、个性化推荐等。有助于提高用户对应用的满意度和粘性

常见推送场景有：微信消息通知栏、新闻推送、外卖状态 等等，我们自身的推送场景有：下载、连线请求、直播提醒 …

核心特点：

- 单向通信：仅服务器->客户端方向
- 基于HTTP：无需特殊协议
- 文本协议：轻量级，易调试
- 自动重连：内置连接恢复机制
- 简单API：浏览器原生支持

#### 传统实时处理方案

- 轮询：客户端定时地向服务端发送请求，询问是否有新数据。服务端只需要检查数据状态，然后将结果返回给客户端。轮询的优点时实现简单，兼容性好。缺点时可能产生较大的延迟，且对服务器资源消耗较高
- 长轮询：轮询的改进版。客户端向服务器发送请求，询问是否有新数据。服务端只需要检查数据状态，然后将结果返回给客户端。如果没有新数据，服务器会等待一定时间（比如30秒超时时间），在这段时间内如果有心数据，就返回给客户端，否则返回空数据。客户端处理完服务器返回的响应后，再次发送新的请求，如此反复。长轮询相较传统轮询减少了请求次数，但仍然存在一定的延迟

相关技术对比：

| 特性       | SSE                 | WebSocket   | 长轮询     |
| :--------- | :------------------ | :---------- | :--------- |
| 通信方向   | 单向(服务器→客户端) | 双向        | 半双工     |
| 协议       | HTTP                | WS/WSS      | HTTP       |
| 数据格式   | 文本                | 二进制/文本 | 任意       |
| 自动重连   | 支持                | 需手动实现  | 需手动实现 |
| 浏览器支持 | 主流浏览器          | 主流浏览器  | 所有浏览器 |

#### 技术实现细节

##### 服务器端实现

```jsx
const http = require('http');

http.createServer((req, res) => {
  if (req.url === '/events') {
    res.writeHead(200, {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache',
      'Connection': 'keep-alive'
    });
    
    // 发送初始数据
    res.write('event: connected\n');
    res.write('data: Welcome to SSE\n\n');
    
    // 定时推送数据
    let counter = 0;
    const interval = setInterval(() => {
      counter++;
      res.write(`data: Server time is ${new Date().toISOString()}\n`);
      res.write(`id: ${counter}\n\n`);  // \n\n表示事件结束
    }, 1000);
    
    // 连接关闭时清理
    req.on('close', () => {
      clearInterval(interval);
    });
  } else {
    res.writeHead(404);
    res.end();
  }
}).listen(3000);
```

##### 相应格式规范

```text
event: messageType\n
data: actual message content\n
id: 123\n
retry: 5000\n\n  // 两个换行符表示事件结束
```

##### 客户端实现

```jsx
const eventSource = new EventSource('/events');

// 通用消息处理
eventSource.onmessage = (e) => {
  console.log('New message:', e.data);
};

// 特定事件类型处理
eventSource.addEventListener('statusUpdate', (e) => {
  console.log('Status update:', JSON.parse(e.data));
});

// 错误处理
eventSource.onerror = (e) => {
  if (e.eventPhase === EventSource.CLOSED) {
    console.log('Connection closed');
  } else {
    console.error('Error:', e);
  }
  // 自动尝试重新连接
};
```

> [!WARNING]
>
> **警告：** 当**不使用 HTTP/2** 时，服务器发送事件（SSE）受到打开连接数的限制，这个限制是*对于浏览器*的，并且设置为非常低的数字（6），打开多个选项卡时可能会特别痛苦。在 [Chrome](https://crbug.com/275955) 和 [Firefox](https://bugzil.la/906896) 中，这个问题已被标记为“不会修复”。这个限制是每个浏览器和域名的，这意味着你可以在所有标签页中打开 6 个 SSE 连接到 `www.example1.com`，以及另外 6 个 SSE 连接到 `www.example2.com`（来源：[Stackoverflow](https://stackoverflow.com/questions/5195452/websockets-vs-server-sent-events-eventsource/5326159)）。当使用 HTTP/2 时，最大并发 *HTTP 流*的数量是由服务器和客户端协商的（默认为 100）。

### Websocket与HTTP的对比

#### 本质区别

| 特性         | WebSocket                      | HTTP                                     |
| :----------- | :----------------------------- | :--------------------------------------- |
| **连接方式** | 持久化全双工连接               | 短连接（HTTP/1.1后默认持久化但非全双工） |
| **通信方向** | 双向通信                       | 单向通信（请求-响应模式）                |
| **协议开销** | 首部字节小（2-10字节）         | 首部较大（约800字节）                    |
| **消息推送** | 服务器可主动推送               | 只能客户端发起请求                       |
| **延迟**     | 极低（无重复握手）             | 较高（每次请求可能需TCP握手）            |
| **适用场景** | 实时应用（聊天、游戏、股票等） | 传统网页、RESTful API                    |
| **协议升级** | 通过HTTP 101切换               | 无协议切换                               |

#### websocket优势

- 真正的全双工通信

  - 服务器可随时主动推送数据（如聊天消息）
  - 客户端可随时发送数据（如游戏操作）
  - 避免HTTP轮询带来的延迟和资源浪费

- 极低的通信延迟

  - 无需重复的TCP握手
  - 避免HTTP队头阻塞

- 更低的协议开销

  - 首部对比：

    ```
    # WebSocket帧（约6字节）
    0x81 0x85 0x00 0x01 0x02 0x03 0x04
    
    # HTTP请求头（约300-800字节）
    GET /chat HTTP/1.1
    Host: example.com
    Connection: keep-alive
    User-Agent: Mozilla/5.0
    Accept: */*
    Cookie: sessionid=1234...
    [更多头部字段...]
    ```

- 更好的带宽利用率

  ```
  // HTTP轮询的无效请求（90%请求返回304）
  // 客户端: 有新数据吗？
  // 服务器: 没有（但消耗了完整请求/响应）
  
  // WebSocket无无效传输
  // 仅当有数据时才传输
  ```

### HTTP1.1 2.0 3.0变化，有哪些缺点

#### HTTP/1.1

##### 核心改进

- 持久连接：默认保持TCP链接（Connection：keep-alive）
- 管道化：允许发送多个请求（但相应必须按顺序返回）
- 分块传输：支持Transfer-Encoding：chunked
- 缓存增强：引入ETag、Cache-Control等头部

##### 主要缺点

| 问题         | 具体表现                             | 影响                         |
| :----------- | :----------------------------------- | :--------------------------- |
| **队头阻塞** | 前一个请求未完成会阻塞后续请求       | 即使管道化也需按序处理响应   |
| **高延迟**   | 每次请求需完整往返                   | 平均增加2-3个RTT时间         |
| **头部冗余** | 每次请求重复发送相同头部（如Cookie） | 浪费带宽（平均800字节/请求） |
| **并发限制** | 浏览器对同一域名限制6-8个TCP连接     | 现代网页需40+资源时严重受限  |

典型性能瓶颈：页面完全加载需要等请求HTML->解析HTML->请求CSS/JS->请求图片->更多资源，之后才能完全加载

#### HTTP/2

##### 核心改进

- 二进制分帧：替代文本格式，更高效解析
- 多路复用：只通过一个TCP连接就可以传输所有的请求数据，解决浏览器同一域名请求数量限制的问题，更容易实现全速传输
- 头部压缩：HPACK算法减少头部体积,并在两端维护了索引表，用于记录出现过的header，后面在传输过程中就可以传输以已经记录过的header键名，对端收到数据后就可以通过键名找到对应的值
- 服务器推送：主动推送关联资源
- 流优先级：指定资源加载顺序

##### 主要缺点

| 问题               | 具体表现                  | 影响                  |
| :----------------- | :------------------------ | :-------------------- |
| **TCP层队头阻塞**  | 单个TCP包丢失会阻塞所有流 | 在丢包率>2%时性能反降 |
| **握手延迟**       | 仍需TCP+TLS握手（2-3RTT） | 移动网络下仍明显延迟  |
| **服务器推送难用** | 缓存一致性难以保证        | 实际使用率低于5%      |
| **部署复杂度**     | 需要ALPN协商              | 旧中间设备可能不支持  |

#### HTTP/3

##### 核心改进

- QUIC协议：基于UDP实现可靠传输
- 0-RTT握手：缓存会话信息实现快速重连
- 改进的多路复用：每个数据包独立加密
- 连接迁移：IP变化时保持连接（如wifi切4G）

##### 主要缺点

| 问题               | 具体表现                  | 影响                       |
| :----------------- | :------------------------ | :------------------------- |
| **UDP被限速**      | 某些网络限制UDP带宽       | 实际速度可能低于TCP        |
| **CPU消耗高**      | QUIC加密计算量增加        | 服务器负载提高约20%        |
| **中间设备兼容性** | 老旧防火墙可能阻断UDP 443 | 连接失败率约1-3%           |
| **部署成本**       | 需要用户态协议栈          | 主流服务器软件支持尚不完善 |

#### 各版本关键指标对比

| 指标               | HTTP/1.1        | HTTP/2  | HTTP/3          |
| :----------------- | :-------------- | :------ | :-------------- |
| **传输层协议**     | TCP             | TCP     | QUIC (over UDP) |
| **握手延迟**       | 3 RTT (TCP+TLS) | 2-3 RTT | 0-1 RTT         |
| **队头阻塞**       | 应用层          | 传输层  | 无              |
| **头部压缩**       | 无              | HPACK   | QPACK           |
| **移动网络适应性** | 差              | 一般    | 优秀            |
| **全球支持率**     | 100%            | ~95%    | ~70% (2023)     |

#### 现实场景中的选择建议

1. **必须使用HTTP/1.1的场景**：
   - 兼容古董设备（如某些IoT设备）
   - 需要通过严格企业防火墙
2. **推荐HTTP/2的场景**：
   - 主流Web网站（最佳兼容性）
   - CDN静态资源分发
   - 需要平衡性能和部署成本的场景
3. **优先考虑HTTP/3的场景**：
   - 高延迟网络（卫星/移动网络）
   - 实时应用（视频会议、云游戏）
   - 频繁切换网络的移动应用
